## 🎯 解決された問題

### 問題1: 2歳戦での戦績データ不足
**Before**: 新馬や2歳馬はデータなしで予測精度低下
**After**: レベル1で「データなし」警告 → レベル2で即座に最新データ取得

### 問題2: 古いデータによる予測精度低下
**Before**: 3ヶ月以上前のデータでも気づかず予測
**After**: レベル1で「データ古い」警告 → 必要に応じてレベル2で更新

### 問題3: 引退馬の誤予測
**Before**: 引退馬も含めて予測し精度低下
**After**: レベル1で「引退の可能性」警告 → ユーザーが判断可能

### 問題4: データベース全体の陳腐化
**Before**: 手動更新のため全体が古くなる
**After**: レベル3で週次自動メンテナンス → データ鮮度維持

## 📊 システム設計の特徴

### 3層アプローチの利点

#### レベル1: クイックチェック（予測時・自動）
- **速度重視**: スクレイピングなし、データベース内検索のみ
- **非侵襲**: 警告表示のみ、データ変更なし
- **ユーザー体験**: 予測を遅らせない（<1秒）

#### レベル2: レース単位更新（手動・必要時）
- **効率性**: 出走馬のみ更新、全体更新は不要
- **即応性**: 1-5分で完了、すぐに予測可能
- **安全性**: 取得のみ、DB反映は手動（将来自動化予定）

#### レベル3: 定期メンテナンス（週次・自動）
- **網羅性**: データベース全体を対象
- **計画性**: 最大50頭/回で段階的更新
- **保守性**: スケジューラで無人運用

### 「歯抜けデータ」問題の回避

ユーザーの懸念: "レース情報取るときに馬のデータを突き合わせて更新すると歯抜けみたいなデータ収集になる"

**解決策**:
- レベル2: 特定レースの馬だけ更新（局所的）
- レベル3: 全体を系統的に更新（包括的）
- 両方を組み合わせることで、必要な馬は即座に更新しつつ、全体の一貫性も維持

## 🚀 使用方法

### 日常的な使用

1. **レースIDを入力** (例: 202512280101)
2. **「予測実行」をクリック**
   - レベル1が自動実行されデータ状態をチェック
   - 警告が出た場合は次のステップへ

3. **警告が出た場合**:
   - `データなし 多数` → 「🔄 このレースのデータ更新」をクリック
   - `データ古い 数頭` → そのまま予測してもOK（精度やや低）
   - `引退可能性` → その馬を除外して判断

4. **予測結果を確認**

### 定期メンテナンス（推奨）

**毎週日曜日の深夜に自動実行**:

```bash
# Windows タスクスケジューラに登録
トリガー: 毎週日曜日 02:00
動作: py C:\...\smart_update_system.py periodic
```

これにより4-5週間で全体のデータが最新化されます。

## 🔧 技術仕様

### データ鮮度の判定基準

| 状態 | 最終出走日 | 処理 |
|------|-----------|------|
| 正常 | 30日以内 | 警告なし |
| 古い | 31-180日前 | 「更新推奨」警告 |
| 非常に古い | 181日以上前 | 「引退可能性」警告 |
| データなし | - | 「データなし」警告 |

### レート制限

- **レベル2**: 1秒/馬（netkeiba への負荷配慮）
- **レベル3**: 1秒/馬、最大50頭/回
