## 🔑 重要な発見

### 発見1: Phase 13の219%回収率は誤り

**証拠**:
```python
# backtest_gui_logic.py MODE B
if MODE == 'B':
    current_date_param = None
    # → predict_core()内で datetime.now() = 2026-02-21
    # → 2024年レース予測時に2026年までのデータを使用
    # → 「後出しジャンケン」状態
```

**影響範囲**:
- `phase13_tansho_all_bands_results.csv`の全結果
- 40-50%確率帯: 219.1%回収率（実際は80-100%程度）
- 全確率帯100%超（実際は一部のみ100%超と予想）

**真の性能予想**:
- 全体回収率: 85-95%
- 最高確率帯: 105-120%
- 低確率帯: 70-85%

**実務的影響**:
- もしPhase 13を実運用していたら大きな損失
- 早期発見により被害を回避
- Phase 2で真の性能を評価する必要あり

### 発見2: 特徴量管理の問題

**問題の構造**:
```
Phase 12: 79特徴量
    ↓
Phase 13: 39特徴量に削減（精査）
    ↓
GUI: Phase 12の79特徴量ロジックのまま
    ↓
モデルファイル: model_phase12_*.pkl（実は Phase 13の39特徴量）
    ↓
結果: 不整合
```

**root cause**:
1. モデルファイルの命名が誤解を招く（phase12 vs phase13）
2. GUIのアップデートが漏れた
3. 特徴量リストのバージョン管理がない

**推奨解決策**:
1. **メタデータ駆動**:
   ```json
   {
     "model_id": "phase13_20260219",
     "feature_list": ["feature1", "feature2", ...],
     "num_features": 39,
     "trained_on": "2020-2023",
     "validated_on": "2024"
   }
   ```

2. **自動検証**:
   ```python
   def load_model(model_path):
       model = pickle.load(model_path)
       metadata = load_metadata(model_path)
       assert len(features) == metadata['num_features']
   ```

3. **命名規則**:
   ```
   model_phase{N}_win_v{YYYYMMDD}.pkl
   model_phase{N}_place_v{YYYYMMDD}.pkl
   model_phase{N}_features_v{YYYYMMDD}.pkl
   ```

### 発見3: データベースの統一が必要

**現状**:
- `netkeiba_data_2020_2024_enhanced.csv`: 英語カラム？（一部）
- `data/main/netkeiba_data_2020_2025_complete.csv`: 日本語カラム
- Phase 13訓練データ: 特定のカラム名前提

**問題**:
- カラム名の不一致により、異なるデータベース間で処理ができない
- 特徴量計算モジュールが特定のカラム名に依存

**推奨解決策**:
1. **統一カラム名規約**:
   ```
   - 着順 → rank
   - 父 → father
   - 母父 → mother_father
   - 調教師 → trainer
   - 騎手 → jockey
   （全て英語で統一）
   ```

2. **自動変換レイヤー**:
   ```python
   COLUMN_MAPPING = {
       '着順': 'rank',
       '父': 'father',
       ...
   }

   def standardize_columns(df):
       return df.rename(columns=COLUMN_MAPPING)
   ```

## 💡 学んだ教訓

### 1. データリークの深刻性

**教訓**:
- 回収率が異常値（150%超）の場合、リークを疑うべき
- 自動検出システムは必須（`check_leakage.py`）
- 「GUIとの一致」を名目にした MODE B は危険

**今後の対策**:
- 全バックテストでリーク検出を強制実行
- CI/CDパイプラインに組み込み
- 異常値検出アラート

### 2. 特徴量管理の重要性

**教訓**:
- モデルと予測コードの特徴量数不一致は致命的
- ファイル名だけでは管理できない（phase12 vs phase13）
- メタデータ駆動の設計が必須

**今後の対策**:
- 特徴量リストをモデルと一緒に保存
- ロード時に自動検証
- バージョン管理システム

### 3. データベース統一の必要性

**教訓**:
- カラム名の不一致は開発効率を著しく低下させる
- 日本語/英語の混在は避けるべき
- 一度統一すれば、以後のメンテナンスが楽

**今後の対策**:
- 統一カラム名規約の策定
- 自動変換レイヤーの実装
- ドキュメント化

### 4. 早期検証の価値

**教訓**:
- Phase 1で重大な問題を2つ発見
- 実運用前に発見できたことは幸運
- 小規模テストの重要性

**今後の対策**:
- 新モデルは必ず小規模検証から
- 異常値が出たら原因究明
- 自動テストスイートの整備
