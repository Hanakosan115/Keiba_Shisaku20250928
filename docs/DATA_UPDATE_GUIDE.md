# スマートデータ更新システム - 使用ガイド

## 📚 3層アプローチの全体像

```
┌─────────────────────────────────────────────────┐
│ レベル1: クイックチェック（予測時・自動）        │
│   タイミング: 予測実行時に毎回                   │
│   処理時間: 1秒未満                             │
│   動作: 警告のみ、更新なし                       │
└─────────────────────────────────────────────────┘
                    ↓ データ古い場合
┌─────────────────────────────────────────────────┐
│ レベル2: レース単位更新（手動）                  │
│   タイミング: ボタンクリック時                   │
│   処理時間: 1-5分（馬数に依存）                  │
│   動作: 出走馬の最新データを一括取得              │
└─────────────────────────────────────────────────┘
                    ↓ 定期的に
┌─────────────────────────────────────────────────┐
│ レベル3: 定期メンテナンス（スケジュール）        │
│   タイミング: 週1回or月1回                      │
│   処理時間: 30分-2時間                          │
│   動作: データベース全体の古いデータを更新        │
└─────────────────────────────────────────────────┘
```

---

## 🔍 レベル1: クイックチェック

### 📋 概要
- **自動実行**: 予測ボタンを押すと自動的に実行
- **超高速**: スクレイピングなし、データベース内チェックのみ
- **警告表示**: 問題があれば通知

### 📊 表示例

```
[2] データ状態チェック
  ℹ 7頭: データなし（予測精度低）
  ⚠ 2頭: データ古い（更新推奨）
  ⚠ 1頭: 引退の可能性あり
    - マルシュロレーヌ (245日前)

  💡 ヒント: メニューから「データ更新」で最新データを取得できます
```

### ✅ メリット
- 予測を遅くしない（1秒未満）
- データの問題を即座に把握
- 更新が必要かどうか判断できる

---

## 🔄 レベル2: レース単位更新

### 📋 概要
- **手動実行**: GUIの「🔄 このレースのデータ更新」ボタン
- **対象**: 現在のレースの出走馬のみ
- **効率的**: 必要な馬だけスクレイピング

### 🎯 使用タイミング
1. レベル1で「データ古い」警告が出た時
2. 2歳戦など新馬が多いレース
3. 重要なレースで最新データが欲しい時

### 📊 実行手順

1. **レースIDを入力**
   ```
   202512280101
   ```

2. **「🔄 このレースのデータ更新」ボタンをクリック**

3. **処理を確認**
   ```
   [1] レース情報取得中... (ID: 202512280101)
     16頭の出走馬を確認

   [2] 一括データ更新開始...

     更新が必要な馬: 7頭

     [1/7] オルフセン (ID: 2023105432)
       最終DB日付: 2024年10月15日
       → 3件の新規レースを発見

     [2/7] フォルテアンジェロ (ID: 2023106891)
       DB: データなし
       → 2件の新規レースを発見

     ...

   更新完了!
     成功: 5頭
     新規なし: 2頭

   💡 注意: データベースへの反映は手動で行ってください
   ```

4. **予測を再実行**
   - データが更新されているので精度向上

### ⚡ 処理時間
- 1頭あたり約1-2秒（レート制限）
- 16頭のレース: 約20-30秒

### ⚠ 注意点
- **現在は取得のみ**: データベースへの自動反映はしない（安全のため）
- 必要に応じて手動でCSVに追記

---

## 🗓️ レベル3: 定期メンテナンス

### 📋 概要
- **スケジュール実行**: 週1回or月1回
- **全体更新**: データベース内の全ての古い馬を更新
- **優先順位**: 最も古いデータから順に更新

### 🎯 使用タイミング
- **推奨**: 毎週日曜日の深夜
- **理由**: 週末レース後に最新データが反映される

### 📊 実行方法

#### コマンドライン実行
```bash
py smart_update_system.py periodic
```

#### 出力例
```
================================================================================
 レベル3: 定期データベースメンテナンス
================================================================================

  対象: 30日以上前のデータ
  更新上限: 50頭/回

  更新対象馬: 234頭
  今回の更新: 50頭

  [1/50] 馬ID 2020105432
    最終: 2024年10月15日 (75日前)
    → 2件の新規レース

  [2/50] 馬ID 2021103891
    最終: 2024年09月28日 (92日前)
    → 新規なし

  ...

  完了: 38/50頭更新
  残り: 184頭
```

#### Windows タスクスケジューラ設定例
```xml
トリガー: 毎週日曜日 02:00
動作: py C:\...\smart_update_system.py periodic
```

### ⏱️ 処理時間
- 50頭更新: 約1-2時間
- 全体(数百頭): 数回に分けて実行

---

## 💡 推奨運用フロー

### 📅 日常的な使用

**1. 予測時（毎回）**
```
レースID入力 → 予測実行 → レベル1が自動チェック
                            ↓
                    警告が出たら判断
```

**2. 警告が出た場合**
```
[データなし 多数] → レベル2で更新 → 再予測
[データ古い 数頭] → そのまま予測OK（精度やや低）
[引退可能性] → その馬は除外して判断
```

### 🗓️ 定期的なメンテナンス

**毎週日曜日（自動）**
```
深夜02:00: レベル3メンテナンス実行
  → 50頭の古いデータを更新
  → 4-5週で全体カバー
```

## 📈 期待される効果

### Before（更新システムなし）
- ❌ データ古い馬も気づかず予測
- ❌ 新馬は全てデフォルト値
- ❌ 引退馬も含まれる
- ❌ 予測精度: 不安定

### After（3層システム導入後）
- ✅ データの鮮度を常にチェック
- ✅ 必要な時だけ効率的に更新
- ✅ 引退馬を警告
- ✅ 予測精度: 向上・安定
